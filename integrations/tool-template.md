# 新工具接入模板

将新的 AI 工具或插件接入 ai-tools-bridge 编排器的标准模板。

---

## 接入步骤

### 1. 创建工具注册文件

在 `integrations/` 目录下创建 `<tool-name>.md`，包含以下结构：

```markdown
# <工具名称>

## 基本信息
- **名称:**
- **用途:** 一句话描述
- **检测方式:** 如何判断此工具已安装（文件路径、CLI 命令等）
- **官网/仓库:** URL

## 工作流阶段映射

此工具在哪个阶段介入？选择一个或多个：

- [ ] Phase 1: 规格生成 — 替代/补充 OpenSpec
- [ ] Phase 2: 深度细化 — 替代/补充 Brainstorming
- [ ] Phase 3: 任务执行 — 替代/补充 Subagent-dev
- [ ] Phase 4: 完成收尾 — 替代/补充 Review+Archive
- [ ] 新阶段 — 需要在工作流中插入新阶段

## 输入

此工具需要读取什么？

- 输入来源（文件路径、数据格式）
- 必需输入 vs 可选输入

## 输出

此工具产出什么？

- 输出位置（文件路径）
- 输出格式
- 下游工具如何消费此输出

## 调用方式

如何调用此工具？

- 命令/技能名称
- 参数格式
- 是否需要用户交互

## 质量门

此工具完成后应满足什么标准？

- [ ] 标准 1
- [ ] 标准 2

## Token 影响

- 预估额外 token 消耗
- 可优化的点

## 冲突处理

如果此工具与已有工具功能重叠：
- 优先使用哪个？
- 是否需要互斥（同一阶段只用一个）？
- 是否可以串联使用？
```

### 2. 更新编排器 SKILL.md

在 SKILL.md 中更新以下部分：

1. **工具检测** — 添加新工具的检测逻辑
2. **阶段映射** — 如果工具对应已有阶段，更新该阶段的描述
3. **新阶段** — 如果需要新阶段，在工作流中插入并更新跳过逻辑
4. **交互格式** — 更新启动时的工具列表和阶段选项

### 3. 测试验证

接入后验证：

- [ ] 工具检测正确（安装时识别到，卸载时识别不到）
- [ ] 工作流在工具不可用时正确降级
- [ ] 与已有工具的串联/互斥逻辑正确
- [ ] 跳过逻辑不受影响
- [ ] Token 消耗在预期范围内

## 示例：接入一个测试覆盖率工具

```markdown
# Test Coverage Analyzer

## 基本信息
- **名称:** test-coverage-analyzer
- **用途:** 自动分析测试覆盖率并生成报告
- **检测方式:** 检查 package.json 中是否有相关依赖，或 CLI 是否可用
- **官网:** https://example.com

## 工作流阶段映射
- [x] Phase 3: 任务执行 — 在每个任务完成后自动运行覆盖率分析
- [x] Phase 4: 完成收尾 — 生成最终覆盖率报告

## 输入
- 测试结果文件（来自 subagent 的测试运行）
- 代码文件路径

## 输出
- 覆盖率报告（终端输出）
- 保存到 openspec/changes/<name>/coverage-report.json

## 调用方式
- 自动触发，不需要手动调用
- 命令: `npm run coverage`

## 质量门
- [ ] 覆盖率 >= 80%
- [ ] 无未覆盖的错误路径

## Token 影响
- 低，只是运行命令和读取结果
- 不需要 subagent

## 冲突处理
- 与 subagent-dev 互补，不冲突
- 在 subagent 完成任务后自动运行
```

## 新阶段模板

如果新工具需要插入一个全新的阶段，使用以下格式定义：

```markdown
## Phase N: <阶段名称>（<工具名称>）

**目标:** 一句话描述

**前置输入:**
- 来自哪个阶段
- 需要什么文件/数据

**执行:**
- 具体步骤

**质量门:**
- [ ] 检查项 1
- [ ] 检查项 2

**完成后提示:**
> Phase N 完成。[产出描述]
> 1. 继续下一阶段
> 2. 暂停
> 3. 跳过
```

新阶段需要更新 SKILL.md 中的：
- 工作流概览图
- 交互格式
- 跳过逻辑表
- 状态文件格式
