# 决策策略指引

明确什么时候 AI 应该自主决策，什么时候必须停下来问用户。

---

## 核心原则

**自主决策的代价低、可逆 → 自主。**
**决策代价高、不可逆、影响范围大 → 问用户。**

---

## 可以自主决策的场景

这些场景 AI 可以直接做决定，不需要停下来问：

- **文件路径选择** — 遵循项目现有目录结构，新文件放在合理的位置
- **命名规范** — 跟随项目中已有的命名模式（camelCase / snake_case / kebab-case）
- **已有 pattern 的跟随** — 项目中已有类似的实现，跟随同样的 pattern
- **错误消息文案** — 只要语义准确，措辞可以自主决定
- **测试数据构造** — 合理的测试数据，不影响真实系统
- **代码格式** — 缩进、空行、import 顺序，跟随项目风格
- **工具选择** — 在已确定的技术栈内选择合适的工具函数（如用 lodash.get 还是可选链）

## 必须问用户的场景

这些场景 AI 必须停下来，提供选项让用户决策：

- **架构变更** — 新增模块、修改模块间关系、改变数据流方向
- **新增依赖** — 引入项目之前没有的第三方库或包
- **影响外部接口的改动** — 修改 API 签名、数据库 schema、配置文件格式
- **多种合理方案并存** — 当有 2+ 个都可行的方案，且各有利弊
- **安全相关决策** — 认证方式、加密算法、权限模型
- **性能取舍** — 以内存换速度、以一致性换可用性等权衡
- **删减功能** — 用户要求的功能在实现中发现不需要或有更好的替代
- **超出 spec 范围** — 实现过程中发现需要做 spec 未涵盖的工作

## 模糊需求处理

当需求描述不够具体时，不要猜测。按以下方式处理：

1. **列出 2-3 个可能的解读**
2. **每个解读附带简短的影响分析**
3. **让用户选择**

示例：
```
需求: "用户登录后跳转到首页"

解读 A: 登录成功后 window.location.replace('/') — 简单直接，丢失浏览器历史
解读 B: 登录成功后 router.push('/') — SPA 内导航，保留历史，推荐
解读 C: 登录成功后显示过渡页"正在跳转..." — 更好的体验，但多一个页面

建议 B。选择？
```

## Phase 2（细化阶段）的特殊策略

细化阶段是发现模糊需求的主要环节。策略：

- **一次发现一个模糊点** — 不要列出 20 个问题让用户一次性回答
- **按影响程度排序** — 先澄清影响架构的问题，再澄清细节问题
- **提供默认选项** — 每个问题都给出推荐答案，用户可以只说"同意"
- **记录已澄清的问题** — 避免重复问同一个问题

## Phase 3（执行阶段）的特殊策略

执行阶段的决策重点是"遇到偏差怎么处理"：

- **小偏差**（代码风格微调、helper 函数选择）→ 自主决定，在 commit message 中说明
- **中偏差**（实现方式与 spec 略有不同但结果等价）→ 继续执行，在阶段结束时汇总报告
- **大偏差**（无法按 spec 实现或需要重大修改）→ 立即暂停，报告给用户

---

## 紧急情况

当 AI 在执行中遇到以下情况，**立即停止并报告**，不需要等阶段结束：

- 发现 spec 中有严重逻辑错误
- 实现会引入安全漏洞
- 会破坏现有功能
- 依赖的服务/库不可用
