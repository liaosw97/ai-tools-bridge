# 决策策略指引

明确什么时候 AI 应该自主决策，什么时候必须停下来问用户。

---

## 核心原则

**自主决策的代价低、可逆 → 自主。**
**决策代价高、不可逆、影响范围大 → 问用户。**

---

## 可以自主决策的场景

- **文件路径选择** — 遵循项目现有目录结构
- **命名规范** — 跟随项目中已有的命名模式
- **已有 pattern 的跟随** — 跟随同样的 pattern
- **错误消息文案** — 只要语义准确
- **测试数据构造** — 合理的测试数据
- **代码格式** — 缩进、空行、import 顺序
- **工具选择** — 在已确定的技术栈内选择合适的工具函数

## 必须问用户的场景

- **架构变更** — 新增模块、修改模块间关系、改变数据流方向
- **新增依赖** — 引入项目之前没有的第三方库或包
- **影响外部接口的改动** — 修改 API 签名、数据库 schema
- **多种合理方案并存** — 当有 2+ 个都可行的方案
- **安全相关决策** — 认证方式、加密算法、权限模型
- **性能取舍** — 以内存换速度、以一致性换可用性
- **删减功能** — 发现需求不需要或有更好的替代
- **超出 spec 范围** — 需要做 spec 未涵盖的工作

## 模糊需求处理

不要猜测，按以下方式处理：

1. **列出 2-3 个可能的解读**
2. **每个解读附带简短的影响分析**
3. **让用户选择**

## 各 Action 的特殊策略

### sdd-brainstorm / sdd-propose（规划阶段）
- 一次发现一个模糊点
- 按影响程度排序
- 提供默认选项
- 记录已澄清的问题

### sdd-code（执行阶段）
- **小偏差**（代码风格微调）→ 自主决定，在 commit message 中说明
- **中偏差**（实现方式不同但结果等价）→ 继续，阶段结束时汇总
- **大偏差**（无法按 spec 实现）→ 立即暂停报告

### sdd-review-*（审查阶段）
- 所有 critical/major issues 必须报告给用户
- minor issues 可以自主标注，不阻断流程

## 紧急情况

立即停止并报告：
- 发现 spec 中有严重逻辑错误
- 实现会引入安全漏洞
- 会破坏现有功能
- 依赖的服务/库不可用
