# 代码质量检查点

每个工作流阶段结束前执行对应检查。不通过时列出具体问题，提供修复/接受风险/暂停三个选项。

---

## Phase 1 质量门（规格生成后）

- [ ] `proposal.md` 存在且描述了明确的意图和范围
- [ ] 至少一个 spec 文件存在
- [ ] `tasks.md` 存在（可以为骨架）
- [ ] 没有重复的 change name
- [ ] specs 描述的是用户可观测的行为，不是内部实现

## Phase 2 质量门（深度细化后）

- [ ] 无 TBD、TODO、"待定"、"后续补充" 等占位符
- [ ] 每个需求有明确的验证标准（怎样算完成）
- [ ] 错误路径和边界条件已覆盖
- [ ] 接口签名（输入/输出/错误码）已明确
- [ ] 无内部矛盾（不同 spec 文件间的描述一致）
- [ ] scope 可控（一个实现周期能完成）
- [ ] 团队规范约束已纳入（从 config.yaml 或项目约定）

## Phase 3 质量门（任务执行后）

对每个完成的任务：

- [ ] 测试通过（有测试，且运行通过）
- [ ] Spec 合规（实现与 spec 描述一致，无遗漏无多余）
- [ ] 无幻觉函数/类型（引用的函数和类型在代码中确实存在）
- [ ] 错误处理已覆盖（不吞错误、不忽略异常）
- [ ] 无硬编码的敏感信息（密钥、密码、内部 IP）
- [ ] 遵循项目现有代码风格

全部任务完成后：

- [ ] 所有 tasks.md 中的任务标记为 `[x]`
- [ ] 整体测试套件通过（不仅是单个任务测试）
- [ ] 无编译/lint 错误

## Phase 4 质量门（收尾时）

- [ ] Code review 无 blocking issues
- [ ] 所有验证命令成功执行
- [ ] CHANGELOG 或 release notes 已更新（如适用）
- [ ] 变更已归档

---

## 常见 AI 编码错误检查

在 Phase 3 执行过程中，对每个 subagent 的产出额外检查：

| 错误类型 | 检查方式 |
|----------|---------|
| 幻觉函数/方法 | grep 验证引用的函数在代码库中存在 |
| 过度设计 | 检查是否引入了 spec 未要求的功能 |
| 遗漏错误处理 | 检查 async/await 是否有 try-catch，I/O 操作是否处理失败 |
| 类型不一致 | 检查函数签名与调用处的参数类型匹配 |
| 硬编码值 | 检查配置项、URL、超时时间等是否提取为常量 |
| 测试遗漏 | 检查边界值（空输入、极大值、null/undefined）是否有测试 |
| 安全漏洞 | 检查用户输入是否经过验证，SQL/命令注入，XSS |

---

## 使用方式

编排器在每个阶段结束时自动调用对应的质量门检查。不需要手动触发。

检查结果格式：
```
质量门检查:
  ✅ proposal.md 存在且非空
  ✅ specs/ 目录有 3 个 spec 文件
  ❌ tasks.md 为空 — 需要至少一个任务

  建议: 运行 /opsx:propose 生成 tasks，或手动补充
```
