# 团队规范检查

面向团队协作的开发规范，在 sdd-brainstorm、sdd-code、sdd-review-* 等 action 中应用。

---

## 规范来源

团队规范从以下位置读取（按优先级排序）：

1. **`openspec/config.yaml`** — 项目级技术栈和约束
2. **项目根目录的 `CLAUDE.md` / `GEMINI.md` / `AGENTS.md`** — AI 工具的项目级指令
3. **`.editorconfig` / `eslint.config.*` / `tsconfig.json`** — 代码风格和类型约束
4. **`docs/` 下的开发文档** — 项目特定的开发规范

---

## 通用规范检查

### 错误处理

- [ ] async 函数有 try-catch 或错误传播机制
- [ ] 不吞错误（空 catch 块）
- [ ] 错误消息具体且有助于调试
- [ ] 外部调用有超时和重试策略
- [ ] 错误边界清晰

### 日志规范

- [ ] 使用项目统一的日志库/方式
- [ ] 日志级别正确
- [ ] 不在日志中输出敏感信息
- [ ] 关键操作有日志记录

### 代码风格一致性

- [ ] 命名风格与项目一致
- [ ] import 顺序与项目一致
- [ ] 注释风格与项目一致
- [ ] 文件组织方式与项目一致

### 接口设计

- [ ] 公共 API 有清晰的输入/输出类型
- [ ] 参数校验在边界处完成
- [ ] 向后兼容性考虑
- [ ] 版本化策略

---

## 安全检查清单

在 sdd-code 每个任务完成后快速扫描：

- [ ] 用户输入经过验证和清理
- [ ] 无 SQL 拼接
- [ ] 无命令注入
- [ ] 无 XSS
- [ ] 敏感数据不暴露在日志/URL/错误消息中
- [ ] 认证/授权检查存在
- [ ] 依赖无已知漏洞

---

## 测试规范

- [ ] 新功能有对应测试
- [ ] 覆盖：正常路径 + 错误路径 + 边界条件
- [ ] 测试独立
- [ ] 测试命名清晰
- [ ] Mock 最小化

---

## 使用方式

**sdd-brainstorm / sdd-propose：**
- 细化 specs 时确保每个需求满足测试规范标准
- 在 design 中体现错误处理和接口设计规范

**sdd-code：**
- 每个任务完成后运行对应检查清单
- 只传任务涉及的规范点给 subagent

**sdd-review-code：**
- Phase 2 审查时运行完整的安全检查清单
