# 团队规范检查

面向团队协作的开发规范，在 Phase 2（细化）和 Phase 3（执行）中应用。

---

## 规范来源

团队规范从以下位置读取（按优先级排序）：

1. **`openspec/config.yaml`** — 项目级技术栈和约束
2. **项目根目录的 `CLAUDE.md` / `GEMINI.md` / `AGENTS.md`** — AI 工具的项目级指令
3. **`.editorconfig` / `eslint.config.*` / `tsconfig.json`** — 代码风格和类型约束
4. **`docs/` 下的开发文档** — 项目特定的开发规范

---

## 通用规范检查

### 错误处理

- [ ] async 函数有 try-catch 或错误传播机制
- [ ] 不吞错误（空 catch 块）
- [ ] 错误消息具体且有助于调试（不只是 "Error occurred"）
- [ ] 外部调用（网络、文件、数据库）有超时和重试策略
- [ ] 错误边界清晰（函数/模块的错误处理职责明确）

### 日志规范

- [ ] 使用项目统一的日志库/方式
- [ ] 日志级别正确（debug/info/warn/error）
- [ ] 不在日志中输出敏感信息
- [ ] 关键操作有日志记录（创建、修改、删除）

### 代码风格一致性

- [ ] 命名风格与项目一致（变量、函数、类、文件）
- [ ] import 顺序与项目一致
- [ ] 注释风格与项目一致（JSDoc/TSDoc/内联注释）
- [ ] 文件组织方式与项目一致

### 接口设计

- [ ] 公共 API 有清晰的输入/输出类型
- [ ] 参数校验在边界处完成（入口函数）
- [ ] 向后兼容性考虑（不随意删除/重命名公共接口）
- [ ] 版本化策略（如果项目有版本管理）

---

## 安全检查清单

在 Phase 3 每个任务完成后快速扫描：

- [ ] 用户输入经过验证和清理
- [ ] 无 SQL 拼接（使用参数化查询）
- [ ] 无命令注入（不直接拼接 shell 命令）
- [ ] 无 XSS（输出经过转义）
- [ ] 敏感数据不暴露在日志/URL/错误消息中
- [ ] 认证/授权检查在需要的地方存在
- [ ] 依赖无已知漏洞（如果项目有依赖审计）

---

## 测试规范

- [ ] 新功能有对应测试
- [ ] 测试覆盖：正常路径 + 错误路径 + 边界条件
- [ ] 测试独立（不依赖执行顺序，不共享可变状态）
- [ ] 测试命名清晰（描述测试的行为，不只是 "test1"）
- [ ] Mock 最小化（只 mock 外部依赖，不 mock 内部模块）

---

## 文档规范

- [ ] 公共 API 有使用文档
- [ ] 复杂逻辑有注释解释"为什么"（不是"做了什么"）
- [ ] CHANGELOG 更新（如果项目维护 CHANGELOG）
- [ ] README 更新（如果新功能改变了使用方式）

---

## 使用方式

**Phase 2（细化阶段）：**
- 细化 specs 时，确保每个需求都满足"测试规范"中的标准
- 在 design.md 中体现"错误处理"和"接口设计"的规范

**Phase 3（执行阶段）：**
- 每个 subagent 完成任务后，运行对应的检查清单
- subagent 的 prompt 中包含相关规范（不传整个文件，只传任务涉及的规范点）

**Phase 4（收尾阶段）：**
- 最终审查时运行完整的安全检查清单
